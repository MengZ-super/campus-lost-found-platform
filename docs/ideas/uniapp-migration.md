# UniApp 替代 Flutter 移动端方案

## Problem Statement

如何用 UniApp 替代 Flutter，以更低的开发和维护成本，为校园失物招领平台交付一套体验良好的移动端 App，并在验证后扩展至微信小程序？

## Recommended Direction

**UniApp + Vue 3 + uni-ui，App 先行，小程序跟进。**

技术栈选择 UniApp + uni-ui 的核心理由是：团队已有 Vue 基础，uni-ui 文档完善、组件覆盖全，能以最低学习成本搭建出中等水准的 UI。后端 API 已经完备，前端管理端的 API 封装和数据模型可以直接复用——UniApp 端本质上只需要"重写 UI 层"。

发布策略调整为 App 先行：先打包 Android/iOS App 做真机验证，避免小程序审核等待期阻塞开发节奏。代码层面从一开始就按 UniApp 规范编写，后续扩展小程序只需处理少量条件编译差异（推送、地图组件等）。

与 Flutter 端的关键差异：地图功能从"纯文本搜索"升级为 UniApp 原生 `map` 组件 + 高德 SDK；推送从 JPush 迁移到 uniPush（App 端）；状态管理从 BLoC 简化为 Pinia。

## Key Assumptions to Validate

- [ ] **uni-ui App 端视觉可接受** — 先搭一个详情页 demo 在真机上看效果，确认"中等水准"是否达标
- [ ] **UniApp 地图组件可用** — 做一个地图选址 POC，测试 `map` 组件在 App 端的定位、搜索、选址流程
- [ ] **图片上传流程通畅** — 测试 App 端相机/相册选择 + 压缩 + multipart 上传的完整链路
- [ ] **uniPush 可替代 JPush** — 验证推送到达率和深链接跳转能力
- [ ] **JWT token 刷新机制可移植** — 用 `uni.request` 封装拦截器，验证 401 自动刷新 + 请求队列重试

## MVP Scope

**In（必须有）：**
- 登录/注册（JWT 认证，token 存储）
- 失物招领列表（分页、搜索、筛选、无限滚动）
- 物品详情（图片轮播、收藏、认领申请）
- 发布/编辑物品（表单、图片上传、地图选址）
- 我的发布（状态管理：关闭/重开/删除）
- 我的认领（查看状态、取消认领）
- 个人资料（查看/编辑）
- 底部 Tab 导航（首页 / 失物招领 / 我的）

**Out（MVP 不做）：**
- 认领审核流程（物主审核认领申请）— 优先级低，后续迭代
- 推送通知 — App 端先不做，小程序阶段再统一接入
- 修改密码 — Flutter 端本身就是 stub
- 微信小程序 — 等 App 验证后再扩展

## Not Doing (and Why)

- **不用 uView Plus** — 文档不够完善，团队 UniApp 经验不足时踩坑成本高，uni-ui 够用
- **不自定义 UI 框架** — 开发量大，中等水准的需求用 uni-ui + 样式微调即可满足
- **不先做小程序** — 小程序审核周期不可控（首次 3-7 天），且需要类目资质，会阻塞开发节奏
- **不保留 Flutter 端** — Flutter 的环境/插件问题不会自行消失，维护两套移动端得不偿失
- **不用 Taro** — 团队不熟 React，引入新框架的学习成本高于复用 Vue 能力

## Open Questions

- **推送方案**：App 端是否真的需要推送？如果需要，uniPush 的免费额度是否够用（日活 < 100 的项目通常够）？
- **App 打包方式**：用 HBuilder 云打包（方便但依赖网络）还是本地打包（需要 Android Studio 环境）？初期建议云打包降低门槛。
- **iOS 发布**：需要 Apple 开发者账号（$99/年），是否有？如果没有，MVP 阶段可以只做 Android。
- **Flutter 端的测试用例**：现有 `test/` 目录下的测试是否需要在 UniApp 端重写？建议只重写核心逻辑测试（API 调用、状态管理），UI 测试可以后补。
